Receiver

Containerized service, per channel

  • Machine-driven paths (POS/ERP API push, webhooks, reverse-API responses): caller authenticates via API key or mTLS per tenant. Open question: machine auth may stay in DIS or move to Customer Master depending on Customer Master's scope. Currently DIS-internal.

02Source Onboarding · assisted

Onboarding flow · system proposes, human approves

DIS UI · Sample upload sub-module

Tenant or Ithina ops

  • Operator (or tenant) uploads a sample file or payload from the source via the DIS UI
  • DIS UI calls the Onboarding Service backend, which stores the sample in GCS under an onboarding-staging path
▼

Onboarding Service NEW

Containerized service + DuckDB

Layer 1 · Schema inference
DuckDB inspects the sample: DESCRIBE SELECT * FROM read_csv('gcs://...'). Returns inferred columns, types, null %, value distributions, example values.
Layer 2 · Mapping & normalization suggestion
Match source columns to canonical columns using exact-name match, fuzzy string similarity, value-pattern heuristics, and historical mappings from similar source types. Also detect format variance from the sample: date format (e.g., DD-MM-YYYY inferred when first pair exceeds 12), decimal separator, boolean and null encodings, enum value sets, timezone (store-local default), units. Output: draft field_map + draft transforms + per-column confidence.
Layer 3 · Validation suggestion
Profile the sample, propose rules: not-null on always-populated columns, numeric range bounds from min/max, regex patterns from values. Output: two draft suites, source-shape (pre-mapping) and canonical-shape (post-mapping), versioned with the mapping.
▼

DIS UI · Onboarding review sub-module

Side-by-side, low-confidence highlighted

  • Proposed mapping shown next to sample data; low-confidence rows highlighted
  • Operator overrides, adds transforms, declares column ownership (required for column-scoped merge)
  • Dry-run against the sample renders a canonical preview
  • Approve → writes to config.source_mappings with status='staged'
▼

Staged rollout · shadow mode

Pipeline runs the staged mapping in parallel; writes to staging schema, not canonical

  • Dataflow detects status='staged' and routes output to the staging.* schema instead of canonical.*
  • Operator reviews shadow output via DIS UI's Onboarding review sub-module (promotion screen)
  • Approve → status='active'; previous active mapping moves to deprecated
  • Reject → status='deprecated'; operator iterates a new version
  • Pipeline picks up the change at next side-input refresh (~30s)

A bad mapping never touches production canonical data. Staging schema is RLS-enforced and is the only data surface affected during shadow rollout.

Suggestion quality · cold-start vs accumulated corpus

Rule-based (v0)

  • ~60-75% columns auto-matched for common source types
  • No corpus needed; ships day one

+ Historical learning

  • ~75-85% as approved mappings accumulate
  • Trigger: after ~20 onboardings

+ LLM-assisted (Vertex AI)

  • ~85-95% for unfamiliar schemas
  • Add only if onboarding throughput becomes a bottleneck

Once a mapping is active, the live data flow below uses it via the pipeline's side-input lookup.

resolve() / validate()
Identity isolation

Tenant / Store Identity Service

Containerized service + cache; mediates access to physically separate admin DB

Interface (gRPC or REST, internal-only)
  • resolve_from_authenticated_context(user_id, tenant_id) → identity · user paths; tenant already authenticated by Customer Master, no token verification done here
  • resolve_from_upload(upload_id) → identity · CSV upload paths (upload_id minted post-auth by DIS UI)
  • resolve_from_endpoint(cfg_id) → identity · reverse-API puller; machine credentials, DIS-internal
  • resolve_from_machine_credential(api_key | mtls_cn) → identity · POS/ERP push paths; machine auth, DIS-internal pending Customer Master scope
  • validate(tenant_id, store_id) → bool · FK-substitute pre-check in the consumer
the resolve_from_token(jwt) interface is not part of the Identity Service. Token verification is Customer Master's job. Identity Service receives an already-authenticated context (user_id, tenant_id) and returns the data-plane identity (store_id, is_active, metadata).
  • Resolve returns {tenant_id, store_id, +metadata} from cache; miss falls through to admin DB
  • Publishes identity.changed on admin DB writes (deferred; Customer Master does not yet emit — Mirror Sync uses DB-pull mode at launch)
  • Stale-while-error caching: on admin DB error, cache continues to serve entries up to 5 min stale. Bounded; counted and alerted.
  • Receiver-local fallback for validate(): the streaming consumer reads identity_mirror directly when Identity Service circuit-breaker is open. Resolve paths cannot fall back (machine credential lookup needs admin DB), but the cache covers steady state.

03Middle · ELT

Streaming consumer

Streaming consumer · containerised service

One service, multi-tenant, multi-source · Pub/Sub pull subscription · specific runtime is a deployment choice. See decisions.md D4 for higher-throughput migration trigger.

  1. Read Pub/Sub message
  2. Fetch chunk (Bronze PG metadata + GCS payload)
  3. Mapping lookup (config.source_mappings, in-process cache refreshed via mapping.changed Pub/Sub topic, 30s steady-state fallback)
  4. Validate tenant/store via Identity Service validate() (with identity_mirror local fallback when circuit-breaker is open)
Step 5 · Pre-mapping validation (source-shape)
Pandera source-shape suite. DataFrame-native (Polars frames). Tenant-readable failure reasons ("expected column item_code, got itemcd"). Linked from DIS UI quarantine console. No compute wasted mapping a malformed chunk.
  1. Apply mapping: rename, normalize, cast, derive. Normalize handles format variance (date formats, decimal separators, timezones, units, enums, boolean and null encodings, casing, whitespace) via declarative transforms in the mapping config, with a named-function escape hatch for source-specific quirks the vocabulary cannot express. Normalization failures route to quarantine with a distinct "normalization failed" reason.
Step 7 · Post-mapping validation (canonical-shape)
Pandera canonical-shape suite. Versioned per (tenant_id, source_id, version). Per-row pass/fail. Failure context links from DIS UI quarantine console show which check failed. Custom checks (cross-field invariants, conditional rules) authored as @pa.dataframe_check functions.
  1. Circuit-breaker on Cloud SQL health : probe SELECT 1 with 100ms timeout before each batch. Unhealthy: route batch to pipeline.dlq Pub/Sub topic; receivers start returning 503 + Retry-After when DLQ depth exceeds threshold.
  2. Branch: valid → sinks · invalid → quarantine
  3. Emit audit event per stage → BigQuery
Compute platform
At beta scale (~150K events/day, ~1.7 events/sec average) the streaming consumer fits on a minimal containerised service with Pub/Sub pull. Transformation logic is factored as pure functions over (mapping, raw_row) → canonical_row; migration to a higher-throughput runtime later is a runner swap, not a rewrite. Migration trigger: sustained 500+ rows/sec for 7 consecutive days. See decisions.md D4.

Cloud SQL Postgres · canonical sink

RLS enforced, no BYPASSRLS

Sink batches rows by tenant_id, opens transaction, runs SET LOCAL app.tenant_id = '...', executes merge upsert + history append, commits.

canonical · hot tables (RLS)
  • current_store_positions · key (tenant_id, store_id, sku_id)
  • 2-3 sibling hot tables (events, store-context state)
  • Column-scoped merge; event-time wins
canonical · event tables (RLS, append-only)
  • store_sku_sale_events, store_sku_change_events
  • Strictly append-only; no UNIQUE constraint. Corrections from sources written as separate rows.
  • Dedup key for "same source event": (tenant_id, store_id, source_id, source_event_id)
  • Latest-wins at read time via window function; reports and dbt views filter to latest. See decisions.md D33.
  • Partitioned by event_date; 35-day Cloud SQL retention (configurable)
  • FK → identity_mirror.stores, identity_mirror.tenants
  • Daily Cloud SQL → BigQuery copy from day 1; partition drop only after retention window
canonical · signal history (RLS)
  • store_sku_signal_history · one row per SKU per as_of_date
  • Append-only; written by daily-compute job; 35-day retention

  • source_mappings (versioned, per tenant+source; status: staged / active / deprecated)
  • expectation_suite_refs NEW

Audit · audit.events

Cloud SQL in Phase 1; BigQuery archive in Phase 3

  • One row per (trace_id, stage)
  • Postgres partition by date (Phase 1); BQ partition by date + cluster by trace_id, tenant_id (Phase 3 archive)
  • Debug: SELECT * WHERE trace_id = '...'
  • Latency SLO computed from commit_ts - received_ts per (tenant, source). Targets: p50 < 3s, p95 < 8s, p99 < 15s. Error budget 5% of p95 in 30-day window.
Quarantine resubmit loop

Resubmit mechanics

Two types, lineage-preserved, rate-limited

Type A · Replay (mapping was fixed)
Tenant or ops clicks "replay" in DIS UI quarantine console. Original GCS payload is re-published to ingress.resubmit Pub/Sub topic. New trace_id with parent_trace_id linking to original. Skips DuckDB pre-flight (chunk already structurally validated). Re-runs against currently active mapping. Outcome appended to audit lineage.
Type B · Fixed file (new payload)
Tenant uploads a corrected file via DIS UI. Treated as a fresh ingress: full pipeline including DuckDB pre-flight, PII redaction, bronze write. Original trace_id linked via parent_trace_id in the new bronze row. Original quarantine row marked resolved_by_resubmit.
  • Rate limit: per tenant, max 100 resubmits per hour. Prevents accidental loops if a mapping is still broken.
  • Resubmit-of-resubmit cap: chain depth max 3. Beyond that, ops must investigate.
  • Visibility: DIS UI shows full lineage (original → resubmits → resolution) by walking parent_trace_id.
  • Pub/Sub topic separation: ingress.resubmit is distinct from ingress.ready so backpressure and metrics are observable independently.

Single HTTP/REST backend for DIS UI. Per-sub-module handlers; one auth integration with Customer Master; one URL the UI calls.

Validates session tokens against Customer Master. Scopes every request by tenant_id from the auth context. Other backends do not authenticate UI users.

Accepts file/payload upload from DIS UI; stores in GCS onboarding-staging path; delegates schema inference + mapping suggestion to the Onboarding Service backend.

Reads/writes mapping drafts in config.source_mappings (status='staged'). Coordinates staged → active promotion via Onboarding Service.

View, edit (new version), deprecate source_mappings rows. Version history. The only write surface to config from the UI path.

Reads quarantine.* tables. Tenant slice (own rows + resubmit). Ops slice (cross-tenant view, replay, mark resolved). Triggers Ithina-side replay via ingress.resubmit Pub/Sub.

Queries Cloud SQL audit.events by trace_id, tenant, store, time range. Read-only. BigQuery archive becomes a secondary read source in Phase 3.
DuckDB query handler (ops only)
Operator pastes GCS bronze URI + SQL; runs in-process via DuckDB. Restricted to ops-role auth claims. Same engine choice as receivers (consistent failure modes).
Data access posture
  • Reads from: Cloud SQL read replica (canonical hot tables for inspection), config schema, quarantine schema, audit schema (Phase 1; BigQuery audit becomes a read source in Phase 3).
  • Writes to: config.source_mappings via mapping CRUD only. Never writes to canonical, history, identity_mirror, or bronze.
  • Calls: Onboarding Service (sample inference, promotion coordination). Does not call Identity Service for auth (Customer Master owns user auth).
  • Auth posture: trusts Customer Master tokens. Validates signature and claims; does not re-resolve identity.
ELT outputs · how Beam writes to its sinks

Valid rows → canonical (single transaction)

Batched by tenant_id, ~500 rows per transaction

BEGIN
SET LOCAL app.tenant_id = '<id>'

-- hot upsert (event-time conditional):
INSERT INTO canonical.<hot> ...
ON CONFLICT (tenant_id, store_id, sku_id)
  DO UPDATE SET ...
  WHERE incoming.event_ts > existing.ts

-- history append:
INSERT INTO canonical.<hist> ...

COMMIT

Hot and history land atomically. FK to identity_mirror enforced. FK failure → retry with backoff → quarantine as last resort.

Invalid rows → quarantine

At-least-once

  • Per-row (post-mapping fail) or per-chunk (pre-mapping fail)
  • Published to quarantine Pub/Sub topic
  • Drainer (containerized service) writes to Cloud SQL quarantine table
  • Idempotent on trace_id + row_hash
  • DIS UI quarantine console reads from the table

Audit → BigQuery (always)

Per stage, per row; not gated by valid/invalid

  • Streaming insert, bundled
  • Partition by date, cluster by trace_id, tenant_id
  • Audit emission failure never blocks canonical writes
  • Gaps re-emittable from bronze on demand

BigQuery canonical_history is NOT written by the streaming pipeline. It is loaded nightly by the batch job (see §04) from the Cloud SQL history slice. This keeps the hot-path latency budget free of BQ write variance.

04Cold path · Analytics offload

Nightly · retail off-hours

Cloud Scheduler → Containerized job

Idempotent, watermark-driven

Steps
  • Run Pandera suite on the slice before BQ load (canonical-shape, batch variant)
  • On suite failure: hold the load, alert ops, do not proceed
  • Load yesterday's history slice → BigQuery canonical_history.*
  • dbt run to refresh derived BQ models (aggregates, denormalised tables for analytics consumers)
  • dbt test with dbt-expectations: freshness, RI, completeness on canonical_history
  • Delete rows with event_ts < now() - 3 months from Cloud SQL (batched)

Full retention, analytics-grade, dbt-modeled

  • Analytics consumers, BI tools, ML feature pipelines
  • Schema evolution managed by Alembic migrations. New canonical columns: Alembic adds with default, backfills from history, mapping configs updated to populate.
