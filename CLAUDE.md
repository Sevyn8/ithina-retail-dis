## What this repo is

DIS (Data Integration System) is a multi-tenant retail data ETL platform on GCP. Tenants submit data via four ingress channels (csv_upload, api, csv_erp, reverse_api). DIS validates, maps, and writes to a canonical Postgres schema (Cloud SQL) and archives long-term to BigQuery.

Beta target: 5 tenants × ~25 stores × ~5000 SKUs (~150K events/day).

## Stack

- **Language:** Python 3.12. Workspace managed by uv (workspace members declared in root `pyproject.toml` once they exist).
- **Web framework:** FastAPI + uvicorn.
- **Database:** Postgres 15 (Cloud SQL). Migrations via Alembic. Async access via SQLAlchemy 2.0 with `[asyncio]`.
- **Validation:** Pandera (decisions.md D4a). Both pre-mapping (source-shape) and post-mapping (canonical-shape) suites.
- **DataFrame layer:** Polars. PyArrow for interop.
- **CSV preview / ad-hoc SQL:** DuckDB.
- **Analytics warehouse:** BigQuery. Models via dbt-bigquery (decisions.md D23).
- **Messaging:** Pub/Sub.
- **Object storage:** GCS.
- **Cache:** Redis (Memorystore in cloud).
- **IDs:** UUIDv7 via `uuid-utils`. Never use `uuid.uuid4`.
- **JSON:** `orjson`. Pydantic v2 uses it as backend.

- **Retry:** `tenacity`.
- **Tracing:** OpenTelemetry. trace_id propagation end-to-end is mandatory (see architecture.md).
- **Logging:** `python-json-logger`. Cloud Logging in cloud targets.

## Hard rules (project-wide invariants)

These hold across every service and lib. Violations are bugs, not preferences.

1. **Tenant isolation via Postgres RLS.** All canonical reads and writes go through `libs/dis-rls`'s session helper. The helper opens a transaction, runs `SET LOCAL app.tenant_id = :id`, and runs every subsequent statement scoped. Never call SQLAlchemy directly against canonical schemas — CI lint forbids it.

2. **PII tokenization at receivers.** Phone, email, loyalty_id, PAN, Aadhaar, and tenant-policy fields are tokenized via `libs/dis-pii` before any persistence step. Receivers are the only place this happens. The tokenization is deterministic HMAC with per-tenant KMS keys (decisions.md D24).

3. **UUIDv7 everywhere.** Every PK and every identifier uses the helper in `libs/dis-core`. Never `uuid.uuid4`.

4. **trace_id is generated by the receiver, propagated end-to-end.** Every Pub/Sub message, audit event, log line, GCS object, bronze row carries it. Never invent a new trace_id mid-pipeline.

5. **Mapping version pinning is mandatory.** Every mapping-produced canonical row (hot + both event tables) carries `mapping_version_id BIGINT NOT NULL`, stamped by the streaming consumer (decisions.md D22). `store_sku_signal_history` is daily-compute output, not mapping-produced, so it has none (D31, D32).

6. **Atomic dual-write.** Streaming consumer writes the hot-table upsert AND the event-table insert in a single Cloud SQL transaction. Never split (decisions.md D30).

7. **Event tables are strictly append-only.** No UNIQUE constraint on event tables. Corrections from sources appear as separate rows. Latest-wins applied at read time via window function over `(tenant_id, store_id, source_id, source_event_id)` (decisions.md D33).

8. **BigQuery access via BqClient only.** All BQ queries go through `libs/dis-core` BqClient which auto-injects `WHERE tenant_id = :tenant_id`. Direct `google-cloud-bigquery` import is forbidden in services. CI lint enforces.

9. **GCS access via dis-storage only.** Canonical path: `tenant/{id}/source/{id}/yyyy=Y/mm=M/dd=D/{trace_id}.{ext}`. Never improvise paths. CI lint forbids direct `google-cloud-storage` import.

10. **Pub/Sub envelopes are frozen contracts.** Schemas in `contracts/pubsub/*.schema.json`. Changing a schema is a coordinated cross-service change, not a single-service edit.

11. **Audit emission is fire-and-forget.** Failures in audit emission are logged but never raise to the caller or block the data path.

12. **No async DB calls without an RLS context.** Every async query must be inside `with rls_session(tenant_id) as s: ...`.

## Code-quality rules

These are not preferences. They're enforced on review.

1. **Plan mode before code.** Every code-writing turn is preceded by plan mode. Operator may waive for trivial single-file edits.
2. **Read before code.** Any unfamiliar service, lib, or schema gets a `view` before any edit. Don't infer structure from filenames.
3. **Tests in the same commit as code.** Every new function/handler/sink/transform ships with a test. "Tests come later" is not a phase.
4. **No silent fallbacks.** `config.get("x", "default")` is forbidden for required values. Required missing → raise a domain error from `libs/dis-core/errors.py`.
5. **Errors carry context.** Every raise / log.error includes `tenant_id`, `trace_id`, and the load-bearing identifier (e.g., `source_id`, `mapping_version_id`). "Failed" with no context is a bug.
6. **No swallowed exceptions.** `except Exception: log.error(...)` without re-raise is forbidden, except in audit emission (Hard rule 11).
7. **One concern per function.** Functions that parse AND validate AND persist get split before merge.

## Build pattern

Slice-driven. Each iteration of work is one slice from `docs/slices/`. The slice doc names: goal, hard constraints, failure-mode categories, plan-mode prompts per checkpoint.

When working a slice:

1. Read the active slice doc in `docs/slices/`.
2. Read the relevant `services/<name>/CLAUDE.md` and `libs/<name>/CLAUDE.md` for that service/lib (auto-loaded when you're in the directory).
3. Use plan mode (Shift+Tab twice) before execution. Plan mode is research/analysis only; it returns a plan you can review.
4. Once the plan looks right, execute against the slice's checkpoints.

When uncertain about anything (architecture, library choice, HTTP shape, idempotency mechanism): ask before coding. Don't guess.

## Conventions

### Imports

- Standard library first; third-party second; first-party (`dis_core`, `dis_canonical`, etc.) last; one blank line between groups.
- No wildcard imports.
- Async-aware modules use `asyncio` / `anyio` patterns. Mixed sync/async in one function is a smell.

### Type hints

- Every function has full type annotations. `from __future__ import annotations` for clean forward references.
- `mypy --strict` should pass on all code (CI will enforce when there's enough code).

### Errors

- Define error types in `libs/dis-core/errors.py`. Don't reach for raw `RuntimeError` or `ValueError`.
- HTTP responses use FastAPI exception handlers that map domain errors to status codes; never `HTTPException` directly in business logic.

### Logging

- Use `structlog`-style context binding via `libs/dis-core` logging helper. Always include `tenant_id`, `trace_id`, `service`, `stage` in log context.
- Never log PII. Never log raw payloads from receivers.

### Tests

- pytest. Async tests via `pytest-asyncio` (asyncio_mode = "auto").
- Test fixtures live in `libs/dis-testing/`. Use the fakes; don't reach for real GCP in tests.
- Unit tests next to code (`services/<name>/tests/unit/`). Integration tests in repo-root `tests/integration/`. e2e in `tests/e2e/`.

## Local devbox

**Facts:**
- Postgres on **port 5433** (Customer Master holds 5432).
- Two Postgres roles: `ithina_dis_admin` (superuser, used by Alembic) and `ithina_dis_user` (NOSUPERUSER NOBYPASSRLS, used by service code).
- Pub/Sub emulator on `localhost:8085`; GCS emulator on `localhost:4443`; Redis on `localhost:6379`.

**Commands:**
- `make run-local` — bring up the stack, create topics, apply migrations.
- `make check` — runs `scripts/check_setup.sh` (6-tier preflight). Run at session start and any time something feels off. Expect 60/60 PASS.
- `make reset-local` — wipe data volumes for a clean slate.
- `make test` / `make lint` / `make format` / `make db-migrate` — daily commands.

**Rule:** if a tool call against the local stack fails (Postgres unreachable, topics missing, GCS down), the first response is `make check`. Don't improvise.

## What NOT to do

- Don't write to canonical tables outside the streaming-consumer or daily-compute. (Daily-compute writes `store_sku_signal_history` and updates derived columns on `store_sku_current_position`, per `decisions.md` D31.)
- Don't generate new `trace_id` outside receivers.
- Don't bypass `libs/dis-rls` for Postgres access.
- Don't bypass `libs/dis-pii` for PII handling.
- Don't import `google-cloud-bigquery` directly. Use `libs/dis-core` BqClient.
- Don't import `google-cloud-storage` directly. Use `libs/dis-storage`.
- Don't add UNIQUE constraints to event tables.
- Don't propose architecture changes mid-slice. Stop, raise, resolve in `decisions.md`, then resume.
- Don't commit before `make check` passes (or `pre-commit` will block you anyway).
